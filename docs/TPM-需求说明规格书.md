## 一、系统定位

通用地理信息协同生产任务管理系统。核心能力：任务全生命周期管理、多业务类型无侵入接入、权限管控、日志审计。当前已对接 DOM 正射影像桥梁去除业务。

设计原则

：

- 公共层（task-management-service）不含任何业务硬编码，新增业务类型不改 TMS 代码
- 外部系统通过注册机制接入，TMS 只管分发和回收，不感知业务细节
- 业务状态→平台状态的映射由外部系统内部完成，TMS 只接收平台标准状态

#### 2.1.2 项目类型、任务分类与层级

##### 2.1.2.1 类型体系总览

系统区分三类独立字典：项目类型（服务领域）、任务类型（技术工序）和计量单位（基本/派生），各自独立管理。计量单位不在类型字典中绑定，而是在创建项目/任务时由用户选择。

```
类型体系
├── 项目类型（project_type_definitions）     ← 仅用于项目/子项目
│   └── 13 种预设服务领域 + 用户可扩展
│
├── 任务类型（task_type_definitions）        ← 仅用于任务
│   ├── 任务类型分组（task_type_group）
│   │   ├── 数据采集（9 种工序）
│   │   ├── 数据处理（9 种工序）
│   │   ├── 产品制作与开发（8 种工序）
│   │   ├── 文档编制（3 种工序）
│   │   └── 其他
│   └── 不绑定计量单位，创建任务时由用户选择
│
└── 计量单位（measurement_unit_definitions）
    ├── 基本计量单位（base_unit_code = null）  ← 仅 system:manager 可管理
    │   └── 5 种预设：米、平方米、立方米、千克、计数
    └── 派生计量单位（base_unit_code ≠ null）  ← 所有用户可创建
        └── 必须关联基本单位 + 换算量（如 1 公里 = 1000 米）
```

约束：

- 项目类型只能用于 PROJECT / PHASE 分类节点（顶层项目或子项目）
- 任务类型只能用于 OPERATION\_TASK / QA\_TASK 分类节点
- 项目类型不允许绑定到任务节点，任务类型不允许绑定到项目节点

##### 2.1.2.6 计量单位体系

计量单位分为**基本计量单位**与**派生计量单位**两级。基本单位是度量基准，派生单位通过换算量与基本单位建立等价关系。

**a. 基本计量单位（base\_unit\_code = null）**

| 编码             | 名称  | 量纲 |
| -------------- | --- | -- |
| UNIT\_M        | 米   | 长度 |
| UNIT\_SQ\_M    | 平方米 | 面积 |
| UNIT\_CUBIC\_M | 立方米 | 体积 |
| UNIT\_KG       | 千克  | 质量 |
| UNIT\_COUNT    | 计数  | 计数 |

**b. 派生计量单位（base\_unit\_code ≠ null）**

| 编码           | 名称   | 基准单位 | 换算量     | 换算说明                 |
| ------------ | ---- | ---- | ------- | -------------------- |

#### 2.1.3 任务状态机

通用任务状态

（TaskStatus 枚举，适用于所有节点）：

| 状态                    | 含义   | 流转                                   | 回退                              |
| --------------------- | ---- | ------------------------------------ | ------------------------------- |
| PENDING               | 待处理  | 创建时默认                                | -                               |
| ASSIGNED              | 待接收  | 指定负责人时                               | ← 撤销指派（→PENDING）               |
| RECEIVED              | 已接收  | 确认接收任务时                              | -                               |
| IN\_PROGRESS          | 进行中  | 开始执行时                                | -                               |
| PAUSED                | 已暂停  | 手动暂停                                 | -                               |
| SUBMITTED\_FOR\_QA    | 待质检  | 操作员点击"提交质检"后                               | ← IN\_PROGRESS（操作员"撤销"，需 assigneeId + `task:execute` 权限）      |
| QA\_COMPLETING        | 质检中  | 质检员点击"接收"后                             | ← IN\_PROGRESS（质检员"不通过"，需 `quality:check` 权限；assigneeId 恢复原操作员）                              |
| QA\_COMPLETED         | 质检完成 | 质检员确认通过时                             | -                               |
| COMPLETED             | 已完成  | 子项目/任务 QA\_COMPLETED 后直接完成；根项目归档后自动完成 | -                               |
| FAILED                | 已失败  | 执行异常                                 | -                               |

项目验收归档阶段

（WorkflowStatus 枚举，仅适用于根项目 PROJECT）：

| 阶段                    | 含义   | 流转方向                | 回退 |
| --------------------- | ---- | ------------------- | -- |
| PENDING\_ACCEPTANCE   | 待验收  | → ACCEPTANCE\_COMPLETED | -  |
| ACCEPTANCE\_COMPLETED | 验收完成 | → ARCHIVED          | -  |
| ARCHIVED              | 项目归档 | 终态（归档后 status 自动设为 COMPLETED） | -  |

> 约束：
>
> - TaskStatus 适用于所有节点（PROJECT / PHASE / OPERATION_TASK），`status` 字段存储；WorkflowStatus 仅适用于根项目 PROJECT，`workflow_status` 字段存储。
> - 除 TaskStatus 表和 WorkflowStatus 表标注可回退的流转外，其余只能向前流转，不可回退、不可跳过。
> - **根项目定义**：没有 `parent_task_id` 的顶层 PROJECT 节点。以下"仅根项目"均指此定义。
> - 子项目和任务节点在 QA_COMPLETED 后直接完成（status=COMPLETED），不经过 WorkflowStatus。
> - **根项目自动流转**：当根项目下所有叶子节点的 QA_COMPLETED 数量 = 叶子节点总数时，项目 workflowStatus 自动设置为 PENDING_ACCEPTANCE（待验收）。由项目创建人确认后流转为 ACCEPTANCE_COMPLETED（验收完成）。由具有 resource:project_archives_save 权限的用户操作后流转为 ARCHIVED（归档）。归档后 status 自动设为 COMPLETED（完成）。

**非叶子节点状态推导规则**：

- 非叶子节点的状态和进度不能直接修改，由其子节点驱动
- 递归统计该节点下所有叶子节点的状态分布数量和叶子节点总数
- 只有全部叶子节点处于同一状态时，才设置该非叶子节点的状态为该状态；否则该节点状态为空
- 看板中非叶子节点不按自身状态分泳道，而是在各状态下显示其直接子节点的数量；某状态下数量为 0 则不显示

#### 2.1.7 叶子节点工作量流转与进度计算

叶子节点（无子任务的节点）采用**瀑布式工作量流转模型**：总工作量按状态依次流转，操作员直接设值各状态工作量，系统自动扣减上游状态。

**a. 状态工作量流转规则**

```
总工作量 W
  ↓ 系统自动（创建时全部量在 PENDING）
PENDING → ASSIGNED → RECEIVED
                       ↓ 操作员录入 IN_PROGRESS 量（增大值从 RECEIVED 扣减）
                    IN_PROGRESS
                       ↓ 操作员累计输入完成量（IN_PROGRESS 量逐次减少）
                       ↓ 完成量 = 总工作量时，点击"提交质检"
                    SUBMITTED_FOR_QA（待质检，全部工作量从 IN_PROGRESS 搬入）
                    ├─ 操作员"撤销" → IN_PROGRESS（工作量退回）
                    └─ 质检员"接收" → QA_COMPLETING（质检中，工作量搬入）
                                         ├─ 质检员"通过" → QA_COMPLETED（工作量搬入）
                                         └─ 质检员"不通过" → IN_PROGRESS（工作量退回，清零完成量，assigneeId 恢复原操作员）
```

| 规则        | 说明                                                                       |
| --------- | ------------------------------------------------------------------------ |
| 初始指派      | 叶子节点创建时，总工作量全部置于 PENDING 状态，`inProgressCompletedWorkload` = 0                                              |
| 系统自动流转    | PENDING → ASSIGNED → RECEIVED 由系统流程自动完成，工作量随状态自动搬移                       |
| 操作员录入     | 操作员在 IN_PROGRESS 状态累计输入完成量，`inProgressCompletedWorkload` 逐次累加；IN_PROGRESS 量逐次减少；完成量 = 总工作量时，"输入完成量"按钮变为"提交质检"按钮 |
| 提交质检      | 点击"提交质检"后，全部工作量从 IN_PROGRESS 搬移到 SUBMITTED_FOR_QA，`inProgressCompletedWorkload` 清零，任务状态 → SUBMITTED_FOR_QA（待质检） |
| 撤销质检      | 操作员点击"撤销"（需 assigneeId + `task:execute` 权限），工作量从 SUBMITTED_FOR_QA 退回 IN_PROGRESS，**保留已填完成量**（`inProgressCompletedWorkload` 不清零）；撤销后再次输入完成量为增量累加，校验累加结果 ≤ 当前 IN_PROGRESS 工作量；仅 SUBMITTED_FOR_QA 状态可撤销 |
| 接收质检      | 质检员点击"接收"（需 `quality:check` 权限），工作量从 SUBMITTED_FOR_QA 搬移到 QA_COMPLETING，assigneeId 转为质检员，原操作员 ID 保存到 `previousAssigneeId` |
| 质检通过      | 质检员点击"通过"（需 `quality:check` 权限，QA_COMPLETING 状态），工作量从 QA_COMPLETING 搬移到 QA_COMPLETED |
| 质检不通过    | 质检员点击"不通过"（需 `quality:check` 权限，QA_COMPLETING 状态），工作量从 QA_COMPLETING 退回 IN_PROGRESS，**清零已填完成量**（`inProgressCompletedWorkload` 清零，返工从头输入）；assigneeId 恢复为原操作员（`previousAssigneeId`） |
| PAUSED 处理 | 暂停不是回退，工作量留在当前状态不动                                                       |
| FAILED 处理 | `FAILED` 仅用于未完成阶段的异常中止，不作为质检不通过路径；`SUBMITTED_FOR_QA` 或 `QA_COMPLETING` 如发生异常，可由具备相应管理权限的用户退回 `IN_PROGRESS` 重新处理；`QA_COMPLETED` 是稳定完成态，不允许直接置为 `FAILED` 或退回 `IN_PROGRESS`，发现问题需新建返修任务并关联原任务 |
| 守恒约束      | 各状态工作量之和必须等于总工作量，瀑布扣减天然保证守恒                                              |
| 进度驱动      | 叶子节点进度由 `inProgressCompletedWorkload × inProgressWeight` 及各状态工作量×权重自动计算，不可直接修改 |
| 外部系统映射    | 外部系统推送总体进度时，进度仅用于生产过程展示：`inProgressCompletedWorkload` = 0，已完成比例映射为 `SUBMITTED_FOR_QA` 工作量，其余量留在 RECEIVED；外部系统推送 `COMPLETED` 时，TMS 将任务置为 `SUBMITTED_FOR_QA`，必须由 TMS 质检员接收并通过后才进入 `QA_COMPLETED` |

**b. 叶子节点进度计算公式**

```
叶子进度 = (IN_PROGRESS_COMPLETED × w_ip + SUBMITTED_FOR_QA量 × 0.95 + QA_COMPLETING量 × 0.95 + QA_COMPLETED量 × 1.0) / 总工作量
```

其中 `IN_PROGRESS_COMPLETED` 即 `inProgressCompletedWorkload` 字段——IN_PROGRESS 状态下已完成但尚未提交质检的工作量。IN_PROGRESS 未完成部分不参与进度计算（隐含权重为 0）。PENDING / ASSIGNED / RECEIVED 状态工作量权重均为 0，公式中省略。

| 状态                            | 权重    | 说明                        |
| ----------------------------- | ----- | ------------------------- |
| PENDING / ASSIGNED / RECEIVED | 0     | 尚未开始，无实际工作产出              |
| IN\_PROGRESS（已完成部分）           | w\_ip | `inProgressCompletedWorkload` 乘以权重参与进度计算；可配置，默认 0.95；项目级设置默认值，子任务继承项目默认值但可单独覆盖 |
| IN\_PROGRESS（未完成部分）           | 0     | `IN_PROGRESS量 - inProgressCompletedWorkload`，权重为 0，不参与进度计算 |
| SUBMITTED\_FOR\_QA            | 0.95  | 已完成待质检确认                  |
| QA\_COMPLETING                | 0.95  | 质检进行中，与提交质检同等权重           |
| QA\_COMPLETED 及之后             | 1.0   | 已确认完成                     |

**c. 非叶子节点进度汇聚**

非叶子节点的进度仍按同质/异质公式由子节点进度汇聚（见 §3.3），但非叶子节点的进度不能直接修改。

**d. 管理看板彩色进度条**

彩色进度条仅用于管理看板（无泳道的视图），不在任务看板的任务卡上显示。非叶子节点在管理看板中显示彩色进度条，颜色段宽度按节点结构类型计算：同质节点按工作量比例，异质节点按子节点权重比例。数量统计仅在点击彩色进度条后的模态框中展示，不参与颜色段宽度计算。

| 颜色 | 包含状态                                 | 含义   |
| -- | ------------------------------------ | ---- |
| 灰色 | PENDING                              | 未指派  |
| 红色 | ASSIGNED + RECEIVED                  | 待接收  |
| 浅绿色 | IN\_PROGRESS（未完成部分）                  | 进行中（未完成）  |
| 深绿色 | IN\_PROGRESS（已完成部分，即 `inProgressCompletedWorkload`） | 可提交质检  |
| 青色 | SUBMITTED\_FOR\_QA + QA\_COMPLETING  | 待质检  |
| 蓝色 | QA\_COMPLETED + COMPLETED + ARCHIVED | 已完成  |

点击彩色进度条弹出模态框，显示各状态的叶子节点数量和工作量分布。顶层项目且 QA\_COMPLETED 叶子数 = 叶子总数时，显示为"待验收"标签而非彩色进度条。

IN_PROGRESS 绿色段细分为两个子段：深绿色（已完成部分，标签"可提交质检"）和浅绿色（未完成部分，标签"进行中"）。

同质节点颜色段宽度按工作量比例计算：

```
同质父节点某颜色段宽度 = Σ(叶子该颜色段工作量_基本单位) / 父节点总工作量_基本单位 × 100%
同质父节点深绿色宽度 = Σ(叶子.inProgressCompletedWorkload_基本单位) / 父节点总工作量_基本单位 × 100%
同质父节点浅绿色宽度 = Σ(叶子.IN_PROGRESS未完成工作量_基本单位) / 父节点总工作量_基本单位 × 100%
```

异质节点颜色段宽度按直接子节点权重比例计算，避免不同计量单位直接相加：

```
异质父节点某颜色段宽度 = Σ(直接子节点该颜色段比例 × 直接子节点weight) / Σ(直接子节点weight)
异质父节点深绿色宽度 = Σ(直接子节点深绿色比例 × 直接子节点weight) / Σ(直接子节点weight)
异质父节点浅绿色宽度 = Σ(直接子节点浅绿色比例 × 直接子节点weight) / Σ(直接子节点weight)
```

其中直接子节点可以是叶子节点，也可以是非叶子节点；非叶子节点先按自身结构递归算出颜色段比例，再参与父节点加权。

**b. 待处理（PENDING）状态**

未指定负责部门/负责人且未指定质检部门/质检员时，任务处于待处理状态。状态卡显示工作量，下方显示操作按钮：

| 按钮 | 所需权限 | 操作结果 |
|------|--------|---------|
| 指派 | 接力棒持有者（controllerId = 当前用户） | 弹出指派表单，指派给其他用户。PENDING/ASSIGNED/RECEIVED 状态均可指派。 |
| 分解 | 接力棒持有者（controllerId = 当前用户） | 弹出分解流程（见下方 d 项），批量创建子项目/子任务。 |

> **未指派 PROJECT 类型可见性**：未指派负责人（assigneeId 为空）的 PROJECT 类型叶子节点，仅创建人可见。其他用户在 my-tree 中不会看到该任务。
>
> **创建/修改添加负责人时状态变更**：PENDING 状态的叶子节点通过创建或修改操作添加负责人后，自动转为 ASSIGNED 状态，工作量从 PENDING 搬入 ASSIGNED，显示在"待接收"泳道。

> **接收按钮可见性规则**（仅在"待接收"ASSIGNED 泳道显示）：
>
> ```
> IF task.type == PROJECT:
>   IF task.assigneeId != NULL:
>     可见 ← (currentUser.id == task.assigneeId) AND currentUser.hasPermission("project:create")
>   ELSE:
>     可见 ← (currentUser.departmentId == task.departmentId) AND currentUser.hasPermission("project:create")
> ELSE:  // OPERATION_TASK 等
>   IF task.assigneeId != NULL:
>     可见 ← (currentUser.id == task.assigneeId) AND currentUser.hasAnyPermission("project:create", "task:create")
>   ELIF task.operatorIds 非空:
>     可见 ← (currentUser.id IN task.operatorIds) AND currentUser.hasAnyPermission("project:create", "task:create")
>   ELSE:
>     可见 ← (currentUser.departmentId == task.departmentId) AND currentUser.hasAnyPermission("project:create", "task:create")
> ```
>
> **接收操作**：工作量从 ASSIGNED→RECEIVED，任务进入已接收状态。**接收操作同时完成接力棒交接**——controllerId 从创建人转移给接收人。所有任务必须先指派再接收，确保指派流程的权限追溯性。
>
> **指派弹窗按类型区分**：
> - PROJECT 类型：标题为"指派项目负责人"，部门标签为"执行部门"。
> - 非 PROJECT 类型：标题为"指派任务"，部门标签为"执行部门"。
> - 人员下拉列表按接力棒持有者权限展示可选用户。
> - 部门选择权限：`project:update_global` 权限的用户可选择任意部门；其他用户只能指派到本部门。
>
> 不具备接力棒持有权的用户：看不到"指派"按钮和"分解"按钮。

**c. 待接收（ASSIGNED）状态**

**d. 分解操作**

分解是将一个叶子任务拆分为多个同质子任务/子项目的操作。权限要求：接力棒持有者（controllerId = 当前用户）。同时拥有 `project:create` 和 `task:create` 权限的用户可在弹窗顶部选择"分解为子项目"或"分解为子任务"。支持 PENDING、ASSIGNED 和 RECEIVED 初始状态。**分解完成后父节点 controllerId 清空（非叶子节点不再需要接力棒）**。流程如下：

| 步骤 | 说明 | 实现状态 |
|------|------|--------|
| ① 输入子任务数量 | 用户输入希望分解的子任务数量（必须 > 0） | ✅ 已实现 |
| ② 均分工作量 | 系统将父任务工作量按数量平均分配到各子任务 | ✅ 已实现 |
| ③ 弹出分配表 | 显示可编辑的任务分配表格，列根据权限动态显示：任务分组+任务类型（仅分解为子任务时显示，必选）、计量单位（自动填写父任务单位，不可更改）、执行部门（`project:update_global` 权限可见，必填）、质检部门（`project:update_global` 权限可见）、项目负责人/执行人（接力棒持有者可见，列头按分解类型自动切换）、质检员（接力棒持有者可见） | ✅ 已实现 |
| ④ 用户调整 | 用户可在分配表中调整各子任务工作量、增加行、删除行 | ✅ 已实现 |
| ⑤ 确认校验 | 确认时校验：各子任务工作量总和必须等于父任务工作量；分解为子任务时同类子任务工作量之和应等于父任务总工作量；子任务名称在同一父节点下必须唯一（不可与兄弟任务重名，也不可彼此重复）；执行部门为必填项；执行部门或执行人相同时界面显示警告提示 | ✅ 已实现 |
| ⑥ 批量创建 | 校验通过后批量创建子项目/子任务（分类和类型与父任务相同，计量单位与父任务相同）；无 `project:update_global` 权限时执行部门和质检部门自动填写为当前用户所属部门，执行人和质检员限定为本部门人员 | ✅ 已实现 |
| ⑦ 取消 | 取消则不做任何操作 | ✅ 已实现 |

> 约束：分解操作为同质分解——子任务的分类、类型和计量单位必须与父任务相同。计量单位自动填写且不可更改。子任务名称在同一父节点下必须唯一（不可与已有兄弟任务重名，也不可彼此重复）。执行部门或执行人被多个子任务共用时界面显示警告提示。人员选择约束：无 `project:update_global` 权限时，执行人和质检人均限定为当前用户所属部门人员，执行部门和质检部门列不显示。

> **指派人与被指派人约束**：指派人（assignerId）和被指派人（assigneeId）不能为同一用户，防止逻辑漏洞。

#### 2.1.10 接力棒控制权模型（v1.13 新增）

接力棒（controllerId）是叶子节点的操作控制权标记。一根接力棒只能传移，不能复制。只有持有接力棒的用户才能执行指派、分解、添加子节点等结构性操作。

**a. 数据模型**

Task 实体新增 `controller_id` 字段（UUID，可空）。非叶子节点（已分解）的 controllerId 为空。

**b. 接力棒生命周期**

```
创建 → controllerId = createdById（创建人自动持有）
  │
  ├─ 指派 → controllerId 不变（仍在创建人手中，等待接收）
  │
  ├─ 接收 → controllerId = 接收人 userId（接力棒交接完成）
  │     │
  │     ├─ 开始处理/输入完成量/提交质检...（不涉及接力棒）
  │     │
  │     ├─ 撤销指派 → controllerId = createdById（归还创建人）
  │     │
  │     └─ 撤销接收审批通过 → controllerId = createdById（归还创建人）
  │
  └─ 分解 → 父节点 controllerId = null（接力已完成，子节点各自持有接力棒）
```

**c. 权限判断规则**

| 操作 | 原判断（v1.12 之前） | 现判断（v1.13） |
|------|---------------------|----------------|
| 指派 | `(project:update_global \|\| department:manager \|\| project:create \|\| task:create) && isCreator` | `isController` |
| 分解 | `department:manager \|\| (PROJECT ? project:create : project:create \|\| task:create)` + `isCreator \|\| admin` | `isController` |
| 添加子节点 | `canUpdate && (canCreateProject \|\| canCreateTask)` | `isController` |
| 编辑 | `createdById == userId`（不变） | 不变 |
| 删除 | `createdById == userId`（不变） | 不变 |

**d. 入口可见性规则**

- **看板任务卡**：只有 `isController` 的用户能看到"指派""分解"按钮
- **项目树右键菜单**："指派""添加子项目""添加子任务"菜单项仅对 `isController` 用户显示
- **分解弹窗内**：执行人/质检员列仅对 `isController` 用户显示

**e. `project:update_global` 与接力棒的关系**

`project:update_global` 不参与接力棒控制——即不能绕过 `isController` 执行指派、分解、添加子节点操作。`project:update_global` 仍可用于：部门范围选择（指派时可选择任意部门）、跨部门可见性（分解时可见所有部门列）等非接力棒控制的操作。`project:update_global` 不参与撤销相关权限判断（撤销指派、审批撤销接收均仅限指派人）。

**f. 编辑/删除独立控制**

编辑和删除操作不受接力棒影响，仍由创建人（createdById）独立控制。交出接力棒后，创建人仍可编辑和删除任务/项目。

**g. controllerId ≠ assigneeId 时的 UI 展示**

当接力棒持有者与当前执行人不同时（典型场景：创建人持有接力棒但已指派给他人），前端需明确区分两者身份：

- **任务卡/详情页**：执行人区域显示 assigneeId 对应用户；接力棒持有者信息仅在操作入口（指派/分解按钮）的 tooltip 中提示"当前由 ××× 持有操作权"
- **接力棒持有者 ≠ 当前用户时**：指派/分解/添加子节点按钮不显示（而非灰显），避免用户困惑
- **assigneeId 为空但 controllerId 非空**（已指派未接收）：任务卡显示"待接收"标签，执行人区域显示"待确认"，不显示指派人姓名

#### 2.2.5 分发回调可靠性规则（冻结）

- 分发请求必须携带幂等键 `dispatchId`（建议 `taskId + timestamp`），外部系统按幂等处理重复请求。
- TMS 在分发后若未收到 `RECEIVED`，按固定策略重试：`最多 3 次`，间隔 `30s / 60s / 120s`（指数退避）。
- 超过最大重试仍未收到 `RECEIVED`：任务标记为"分发异常待人工处理"，并触发告警。
- 所有回调（含 `RECEIVED` 与后续状态）必须记录 `requestId` 与回调时间，便于审计追踪。

**b. 接力棒（controllerId）权限覆盖规则**

> **核心原则**：指派、分解、添加子节点等结构性操作，仅接力棒持有者（`controllerId == userId`）可以执行。其他所有权限（`project:update_global`、`department:manager`、`project:create`、`task:create`等）不再参与这些操作的权限判断。
>
> **不受接力棒影响的操作**：编辑和删除仍由创建人（createdById）独立控制；开始处理、输入完成量、提交质检、质检通过/退回等执行类操作不受接力棒影响。
>
> **`project:update_global`**：不参与接力棒控制，也不参与撤销相关权限判断。可用于部门范围选择（指派时可跨部门）、跨部门可见性（分解时可见所有部门列）等操作。

**c. 权限来源**

所有权限由 SSO/UPM 统一管理，TMS 本地不维护任何用户和角色。SSO 不可用时的本地用户不拥有任何 TMS 操作权限，仅可访问 BRS 本地功能。

> **前端导航权限控制**：管理看板入口和设置页面（项目类型、任务类型、计量单位）仅对拥有 `project:read_*` 或 `task:read_*` 权限的用户可见；无上述权限的用户（如仅有 `task:execute` 权限的操作人员）登录后默认进入任务看板，不显示管理看板和设置导航项。

### 3.4 进度汇聚规则

层级深度限制

：任务树最深不超过 5 层（根节点 PROJECT 深度为 0，叶子节点最大深度为 4）。创建子任务时校验深度，超出上限返回错误。

直接子结构约束：

- 同一父节点的直接子节点类型要么全相同（同质），要么全不同（异质）；适用于所有层级（项目/子项目/任务）
- 兄弟项目/子项目/任务的子节点类型允许相同，不强制同类型归集到同一目录
- 父节点的 compositionMode 由直接子节点实时判定并缓存

叶子节点规则

：

- 叶子节点（无子任务的节点）采用瀑布式工作量流转模型（详见 §2.1.7）
- 叶子节点进度由各状态工作量×权重计算：`进度 = (inProgressCompletedWorkload × w_ip + SUBMITTED_FOR_QA量 × 0.95 + QA_COMPLETING量 × 0.95 + QA_COMPLETED量 × 1.0) / 总工作量`（w_ip 默认 0.95；IN_PROGRESS 未完成部分权重为 0，不参与计算）
- 关联外部系统的叶子节点，其状态和进度由外部系统推送更新；否则由相应权限的用户手动设置
- 外部系统推送总体进度时，进度仅用于生产过程展示：已完成比例映射为 SUBMITTED\_FOR\_QA 量，其余量留在 RECEIVED，`inProgressCompletedWorkload` 固定为 0；外部系统推送 COMPLETED 时，任务进入 SUBMITTED\_FOR\_QA，由 TMS 质检员接收并通过后才进入 QA\_COMPLETED

非叶子节点规则

：

- 非叶子节点的状态和进度不能直接修改，由其子节点驱动
- 非叶子节点状态推导：递归统计叶子节点各状态数量，全部叶子同状态才设父状态，否则状态为空
- 顶层项目自动流转：质检完成叶子数=叶子总数→PENDING\_ACCEPTANCE（待验收）→ACCEPTANCE\_COMPLETED（验收完成，创建人确认）→ARCHIVED（归档，resource:project\_archives\_save 权限）→COMPLETED（完成，自动）。**仅根项目（无 parent\_task\_id 的顶层 PROJECT）存在此流转链，子项目（PHASE）和任务节点不经过这四个状态**

父节点汇聚规则

：

- 同质任务：`父进度 = Σ(子进度 × 子权重 × 子工作量_基本单位) / Σ(子权重 × 子工作量_基本单位)`，其中 `子工作量_基本单位 = 子工作量 × 换算量`（基本单位换算量为 1，派生单位按 conversion\_factor 换算）
- 异质任务：`父进度 = Σ(子进度 × 子权重) / Σ(子权重)`
- 子任务 `weight` 取值范围 `0.01~100`，未设置时按 `1` 参与计算
- 同质任务父节点填写工作量时，直接子节点工作量（换算为基本单位后）之和必须等于父节点工作量（换算为基本单位后）。不一致时必须提示差额，提示内容至少包含：子节点合计、父节点工作量、差额、建议调整父节点或子节点工作量。工作量为空或 ≤0 时创建/更新报错。父节点未填写时由子节点动态汇总（汇总值以基本单位为准）

```
项目 P（PROJECT, type=工程测量, 单位=平方公里, workload=100, compMode=HOMOGENEOUS）
 ├── 子项目 Phase1（PHASE, type=工程测量, 单位=平方公里, workload=60, 进度=80）
 │    ├── 任务 A（OPERATION_TASK, type=外业调绘, 单位=平方公里, weight=1, workload=30, 进度=90）
 │    └── 任务 B（OPERATION_TASK, type=外业调绘, 单位=平方公里, weight=1, workload=30, 进度=70）
 └── 子项目 Phase2（PHASE, type=工程测量, 单位=平方公里, workload=40, 进度=50）
      ├── 任务 C（OPERATION_TASK, type=外业调绘, 单位=平方公里, weight=1, workload=20, 进度=60）
      └── 任务 D（OPERATION_TASK, type=外业调绘, 单位=平方公里, weight=1, workload=20, 进度=40）

Phase1 进度 = (90×1×30 + 70×1×30) / (1×30 + 1×30) = 80
Phase2 进度 = (60×1×20 + 40×1×20) / (1×20 + 1×20) = 50
P 进度     = (80×1×60 + 50×1×40) / (1×60 + 1×40) = 68
工作量校验（基本单位）：Phase1(60) + Phase2(40) = P(100) ✓

── 含派生单位的示例 ──

项目 P（PROJECT, type=工程测量, 基本单位=米, workload=100000, compMode=HOMOGENEOUS）
 ├── 任务 A（单位=公里, 换算量=1000, weight=1, workload=60, 进度=90）
 └── 任务 B（单位=米, 换算量=1, weight=1, workload=40000, 进度=70）

A 工作量_基本单位 = 60 × 1000 = 60000
B 工作量_基本单位 = 40000 × 1 = 40000
P 进度 = (90×1×60000 + 70×1×40000) / (1×60000 + 1×40000) = 82
工作量校验（基本单位）：60000 + 40000 = 100000 ✓（不等于时必须提示差额；空/≤0 时报错）
```

## 四、系统间交互

### 4.1 交互协议

| 交互       | 方向             | 内容                                                                      |
| -------- | -------------- | ----------------------------------------------------------------------- |
| 系统注册     | 外部系统 → TMS     | ssoClientId + supportedTaskTypes + callbackPath + dashboardUrl          |
| SSO 登出绑定 | 外部系统 → SSO     | 首次登录后自行绑定登出回调（TMS 不参与）                                                  |
| 项目分发     | TMS → 外部系统     | 项目基本信息（内网无认证）                                                           |
| 外部推送     | 外部系统 → TMS     | 只接收顶层项目，不接收子任务（子任务由 TMS 内部创建）                                           |
| 接收确认     | 外部系统 → TMS     | 回调 RECEIVED 确认已接收分发任务                                                   |
| 状态同步     | 外部系统 → TMS     | 仅平台标准生产状态（PENDING/ASSIGNED/RECEIVED/IN\_PROGRESS/PAUSED/COMPLETED/FAILED）；外部 COMPLETED 映射为 TMS `SUBMITTED_FOR_QA`，不直接映射为 `QA_COMPLETED`；TMS 质检通过后才进入 `QA_COMPLETED` |
| 执行结果     | 外部系统 → TMS     | results JSON 写入 outputResults                                           |
| 任务完成数据   | 外部系统 → TMS     | completion-data（含阶段责任人、工作量）                                             |
| 人员工作统计   | TMS → 外部系统     | personnel-stats（可选，第三方应用可自行实现）                                          |
| 业务详情查看   | TMS 用户 → 第三方应用 | 通过 dashboardUrl 跳转，SSO 互通免登录                                            |

### 4.2 不交互的内容

以下由外部系统独立处理，TMS 不感知：

- 具体业务操作步骤（如桥梁定位、掩膜生成、影像修复）
- 业务状态语义（如"待初检""初检通过"等中文状态）
- 业务算法执行
